01 · Why

왜 직접 만들게 됐을까요

CS와 품질관리는 매일 새로운 데이터가 쌓이는 업무입니다. 하지만 그 데이터를 보고서로 만드는 과정, 그리고 실무에서 반복되는 판단(부품 원인 분석, 견적 산출, 회수 안내 등)은 사람이 매번 손으로 처리해야 했습니다.

Before
  • 여러 시트·문서에 흩어진 CS·품질 데이터
  • 반복되는 보고서 작업, 매번 새로 정리
  • 부품 원인·견적 산출은 경험에 의존
  • 팀원 간 업무 방식·툴이 제각각
→
After
  • 하나의 대시보드로 CS·품질 데이터 통합
  • 보고서 자동 집계 + 정해진 양식으로 발행
  • 과거 데이터 기반 자동 원인 추천 · 견적 계산
  • 팀 공통 Tool 허브로 업무 표준화
02 · How it's built

전체 구조는 4단계로 흐릅니다

용어 옆 ? 표시에 마우스를 올리면 설명이 뜹니다. (발표 중 청중 질문에 즉석 대응용)

01 DATA
노션·채널톡은 API로 직접 연동하고, 연동이 어려운 사이트는 CSV 다운로드? CSV 방식스마트스토어처럼 API가 막혀 있거나 고정 IP가 필요한 곳은 구글시트에서 CSV로 내려받아 data/ 폴더에 넣고, 페이지가 이를 직접 읽어들이는 방식으로 우회합니다. 후 data 폴더에 적재해 불러옵니다.
↓
02 BRIDGE
모든 페이지는 GAS? Google Apps Script구글이 제공하는 무료 서버리스 백엔드. 별도 서버 없이 구글시트·계정 권한만으로 API처럼 데이터를 주고받을 수 있어, 이 시스템의 중앙 통신 창구 역할을 합니다. 를 거쳐 데이터를 주고받습니다. 무거운 데이터는 매일 새벽 자동 트리거로 JSON 스냅샷? 캐싱 전략변하지 않는 고정값(과거 이력 등)은 매일 1시 GAS 트리거로 미리 JSON 파일에 굳혀두고, 페이지는 이 JSON만 읽습니다. 매번 실시간 조회하지 않아 로딩 속도가 크게 빨라집니다. 으로 미리 굳혀 로딩 속도를 최적화합니다.
↓
03 FRONT
역할 기반 권한? 접근 제어관리자 / 리더급 / 꿈비 / 계열사 4단계로 로그인 계정을 나누고, 역할별로 볼 수 있는 메뉴·데이터 범위를 다르게 제어합니다. 을 가진 허브 화면(index.html) 하나에서 주간·월간 보고서, 실무 Tool, 팀 채팅, 교육자료, 지식DB까지 모두 iframe으로 연결됩니다.
↓
04 DOCS
시스템 구조 자체를 설명하는 관리 매뉴얼 페이지가 별도로 존재해서, 파일이 추가·변경될 때마다 문서도 함께 갱신되는 규칙으로 운영합니다. 즉, 시스템이 스스로를 설명합니다.
03 · Journey

이렇게 확장해왔습니다

처음부터 큰 그림을 그리고 시작한 게 아니라, 필요한 것을 하나씩 만들다 보니 지금의 구조가 되었습니다.

STEP 1
주간·월간 보고서 자동화부터 시작
반복되던 수기 보고서 작업을, 정해진 디자인 양식의 HTML 보고서로 표준화. GitHub Pages에 쌓아가는 구조를 잡음.
STEP 2
품질관리 대시보드 확장
불량현황·반품현황·처리현황 등 품질 지표를 별도 화면으로 세분화, 실패모드(수분유입 vs 뚜껑누수 등)를 구분해 관리.
STEP 3
실무 Tool 12종으로 업무 자동화
AS 기술진단·부품원인 추천·유상견적 산출·회수문구 생성 등, 반복되는 실무 판단을 자동화한 팀 전용 Tool 허브 구축.
STEP 4
팀원 대상 AI 교육 13회차 진행
"AI로 AI 활용하기"부터 "GAS 트리거 자동화"까지, 실습 위주로 팀원들이 직접 자신의 업무 Tool을 만들어보는 교육 운영.
STEP 5
전사 지식DB 프로그램까지 확장
팀 업무를 넘어, 전사 지식공유 프로그램(조편성·우수사례·수상작 관리)을 자동화 시스템으로 운영.
